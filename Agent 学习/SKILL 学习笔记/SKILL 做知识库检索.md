我的理解：通过 SKILL 教会 AI 使用 grep 命令在文件系统中进行检索。参考 Codex。

# 功能
1. 渐进式检索：少读但读准，按需扩展，渐进式披露。
2. 保持简单可控：用户告诉知识库在什么地方，其他事情 SKILL 自主完成。

## 检索要求
1. 指定目录检索
2. 跨格式联合查询
3. 渐进式披露
	1. 关键词缩小范围
	2. 精准打开少量文件
	3. 快速定位问题
	4. 按问题组织成果
		- 结合上下文，给出可用的回答
	5. 复杂问题，多轮查找
		1. 锁定核心文件
		2. 按章节检索
		3. 组合回答

## 设计方法：
1. 定位领域：分析问题 => 锁定文件夹
2. 定位文件：使用 grep 筛选出可能的相关文件
3. 定位内容：根据不同文件，设置不同的策略进行定位
	- Markdown/文本：直接定位解析
	- PDF：编写代码调用能力，按页、章节内容
	- Excel：只看跟问题相关的表、行、列

## 文件系统设计：
1. 领域划分：文件夹按照领域划分，仅包含特定领域内容。
2. 目录所有：文件夹下有目录索引文件，描述当前文件夹下，每个文件的用途，让 AI 快速定位文件。
3. 策略加载：针对不同的文件类型 ，设置不同的检索策略。

#### 小方法：
1. PDF 转 TxT
2. 文件尽量以纯文本

# 问题
1. 首次检索效率：针对非文本文件，首次检索效率低。
	- 可以通过预处理文件处理
2. 调用稳定性：多轮对话检索，AI 可能忘记调用 SKILL。
3. Token 消耗：多轮检索 Token 消耗增加

# 进阶

通过已有知识库内容，用 skill-creator skill 进行创建：

> [!quote]
> 当前项目是一个文档站（包含多个 md/mdx 文件），我想创建一个专用于知识库检索的 skill，它可以替代传统的 RAG，用于当前文档站的知识检索，你帮我创建做广告 Skill（用中文）。Skills 的基本思路是：先根据用户的问题，提取出需要检索的关键字，然后使用 grep 命令找到对应的上下文，然后分析上下文，如果检索到的信息和问题不符合，需要继续检索其他可能关键字，最大重试五次。
