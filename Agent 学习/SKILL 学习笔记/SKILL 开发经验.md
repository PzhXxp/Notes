# Code 辅助 skill 开发经验

将需求分析，方案涉及，代码落地，评审验收拆解开。

定制代码 skill，目前更新了三个版本，第一版所有事情放到一个 skill 上，问题是流程复杂，轮次一多容易漂移。

开发 skill 考虑几点：

1. .skill 的类型：步骤形？说明形？清单形？
2. skill 必须遵守的条件。
3. 边界行为，例如搜索，要限定搜索范围，必要时限定轮次
4. 让 skill 给出选择，而不是唯一方案。

模型比想象的聪明，能理解你的意图，但是也比想象中更容易发散，写步骤型 skill 的时候，不要限定很死，否则模型很容易越界。

当做很复杂的系统的时候，例如定制方案评估，不要把每个方案的界限限定得很死，写大几千的说明，大部门模型执行 skill 的是时候，skill 主体，引用文件，对话内容，几轮下来，上下文很容易被压缩，导致漂移。

既然已经让 ai 去做，就要给它主导权，我们限定好边界条件，评估标准，其他的交给 ai，它比人想得更聪明。如果担心 ai 乱来，这正式评估（评审）环节的作用。

多个 skill 可以串联，但是考虑复杂任务多个 skill 串联执行容易导致 context 压缩，所以每个 skill 结束都要输出有效文档，能让 skill 根据文档重新开始。或者在 context 被压缩，有效信息不足的时候，通过文档开始新的干净的对话。
